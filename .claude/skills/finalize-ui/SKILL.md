# finalize-ui

ユーザーが承認した apps/web のモックを、設計書（README.md）の最終 UI 仕様として落とし込む skill。

## 前提

- このskillの前に `design-feature` skill が完了している（README.md と モックが存在する）
- ユーザーが「OK」「fix」「これで確定」とモックを承認している
- エンジニアによるスタイル調整も完了している

## 目的

確定したモックを「**画面設計書**」と「**デザイン設計書**」として README.md に記録し、後から実装する人（人間 / AI）が UI を再現できる状態にする。

**目的ではないこと**:
- コンポーネントの細かい実装手順を書く（それは step ファイルの役割）
- 完璧なStorybook相当の文書化（過剰）
- スクリーンショット添付（テキストで再現できる粒度を目指す）

## 進め方

### Step 1: 対象機能とモックを特定する

ユーザーに確認する（もしくはコンテキストから判断する）:
- どの機能の UI を確定するか（例: matching、profile）
- 対象のモックファイル（例: `apps/web/src/app/{feature}/page.tsx`）

### Step 2: 確定したモックを読み込む

対象機能の以下を全て読む:
- `apps/web/src/app/{feature}/**/*.tsx`
- 関連するコンポーネント（`apps/web/src/components/...` の中で使用しているもの）
- `apps/web/src/app/globals.css`（カスタムスタイル）
- `apps/web/src/libs/mock-data.ts`（参照しているダミーデータ）

### Step 3: 既存の設計書を読む

`docs/spec/{feature}/README.md` を読み、既存の「UI 設計」セクションがあれば把握する。

### Step 4: UI 設計セクションを更新する

README.md の「UI 設計」セクションを更新する（既存があれば置き換え、なければ追加）。

#### 画面設計書（記録する内容）

**画面一覧（テーブル）**:
| パス | 画面名 | 認証 |

**各画面ごとに**:
1. **ASCII ワイヤーフレーム**: 確定したレイアウトを ASCII で表現
2. **状態遷移**: 状態が複数ある場合（待機 / 成立 / 通話中 等）はそれぞれワイヤーフレームを描く
3. **要素の説明**: 各 UI 要素の役割・動作・表示条件を箇条書き
4. **インタラクション**: ボタン押下時の挙動、遷移先

#### デザイン設計書（記録する内容）

確定モックから読み取り、文章化する:
1. **カラーパレット**: プライマリ / セカンダリ / アクセント / 背景 / テキスト（CSS変数または Tailwind クラスで）
2. **タイポグラフィ**: 見出し / 本文 / キャプション のサイズ・ウェイト
3. **コンポーネント**: ボタン / カード / バッジ などの基本スタイル
4. **アニメーション**: ホバー / トランジション / 特殊効果（グロー、紙吹雪、グラデーション等）
5. **ダークモード対応**: 必要なら明記
6. **レスポンシブ**: ブレークポイントごとの挙動

### Step 5: 既存形式に合わせる

`docs/spec/matching/README.md` の「UI 設計」セクションが最も網羅的な参考例。形式を合わせる:
- 表（パス・画面名・認証）
- ASCII ワイヤーフレーム（コードブロック）
- 「###」見出しで画面ごとに分割
- 動作仕様は箇条書き（太字でラベル + 説明）
- mermaid 図を使う場合もあり（状態遷移図など）

### Step 6: 目次を更新する

CLAUDE.md のルールに従い、README.md の目次（Table of Contents）を新しい見出しに合わせて更新する。
- GitHub Markdown アンカーリンク形式（例: `[UI 設計](#ui-設計)`）
- ネストしたリスト形式

### Step 7: ユーザーに差分を報告

更新内容のサマリ（追加された画面、デザイン仕様の項目）を簡潔に報告する。
ファイルパス + 行数で具体的に示す。

## ASCII ワイヤーフレームの書き方

`docs/spec/matching/README.md` のサンプルを参考にする:

```
┌─────────────────────────────────────────┐
│ [=== テーマ残り 18s ===============---] │ ← テーマごとタイマー
│ ┌─ シャボン玉風オーバーレイ ─────────┐ │
│ │  好きな食べ物のジャンルは？         │ │ ← トークテーマ
│ │  [🍣 和食] [🍝 洋食] [🍜 中華]    │ │ ← 選択肢テーマ
│ └────────────────────────────────────┘ │
└─────────────────────────────────────────┘
```

- 矢印 `←` で各要素の役割を注記
- 状態が変わる部分は別ワイヤーフレームで分けて描く
- アスキーが破綻しそうな複雑な画面は、テキスト箇条書きで補完する

## やってはいけないこと

- モックの実装コードをそのまま README に貼り付ける（仕様書はコードではない）
- step ファイルに UI 仕様を書く（README の役割）
- スクリーンショット画像を新規生成して保存する
- ユーザーが OK していない状態で仕様化する
- 既存の matching/README.md の形式から逸脱する
- 目次の更新を忘れる
